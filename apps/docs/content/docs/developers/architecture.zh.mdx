## Overview

Multica is a Go backend plus a pnpm/Turborepo frontend monorepo. It is local-first and uses a solo local user.

## Project Structure

| Directory | Purpose |
| --- | --- |
| `server/` | Go backend, CLI, daemon, migrations, sqlc queries |
| `apps/web/` | Next.js app |
| `apps/desktop/` | Electron app |
| `apps/docs/` | Fumadocs site |
| `packages/core/` | API clients, stores, query options, shared types |
| `packages/ui/` | Atomic UI components |
| `packages/views/` | Shared business UI |

## Backend

- `cmd/server` - HTTP and WebSocket server.
- `cmd/multica` - CLI entry point.
- `internal/daemon` - local daemon that executes agent tasks.
- `internal/handler` - domain handlers.
- `internal/realtime` - WebSocket hub.
- `pkg/db/queries` - sqlc query definitions.
- `migrations` - PostgreSQL migrations.

The server injects the solo local user for app and daemon routes. There is no JWT, OAuth, cookie auth, personal access token, or invitation layer.

## Frontend

- TanStack Query owns server state.
- Zustand owns client state.
- WebSocket events invalidate query caches.
- Shared views use `NavigationAdapter` and never import Next.js or React Router directly.

## Repositories and Runtimes

Workspaces can reference local repository paths. Remote repository URLs are rejected.

Agent execution happens through local runtimes registered by the daemon. The daemon detects installed CLIs and reports them to the server.
